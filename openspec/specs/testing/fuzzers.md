# Fuzzers

**Crates**: `canister-threshold-sig-fuzz`, `ic-crypto-tree-hash-fuzz`, `ic-crypto-tree-hash-fuzz-check-witness-equality-utils`, `ic-crypto-internal-csp-proptest-utils`, `ic-validator-fuzz`, `management-canister-types-fuzz`, `wasm_fuzzers`

The `rs/fuzzers/` directory contains fuzz testing targets for critical IC components. These use `libfuzzer` via `libfuzzer-sys` to discover bugs through randomized input generation.

## Requirements

### Requirement: Bitcoin Deserialization Fuzzers

Fuzz targets in `rs/fuzzers/bitcoin/fuzz_targets/` test Bitcoin data structure deserialization robustness.

#### Scenario: Fuzz Bitcoin block deserialization
- **WHEN** arbitrary bytes are provided to the `deserialize_bitcoin_block` fuzz target
- **THEN** it attempts to deserialize the bytes as a `bitcoin::Block`
- **AND** if deserialization succeeds, the block is re-serialized
- **AND** the re-serialized bytes are asserted to be identical to the input (round-trip property)
- **AND** if deserialization fails, the input is rejected from the corpus

#### Scenario: Fuzz Bitcoin transaction deserialization
- **WHEN** arbitrary bytes are provided to the `deserialize_bitcoin_transaction` fuzz target
- **THEN** it attempts to deserialize the bytes as a Bitcoin transaction
- **AND** verifies the round-trip serialize/deserialize property

#### Scenario: Fuzz Bitcoin raw network message deserialization
- **WHEN** arbitrary bytes are provided to the `deserialize_bitcoin_raw_network_message` fuzz target
- **THEN** it attempts to deserialize the bytes as a Bitcoin raw network message
- **AND** verifies serialization round-trip consistency

### Requirement: Candid Fuzzers

Fuzz targets in `rs/fuzzers/candid/fuzz_targets/` test Candid encoding/decoding robustness.

#### Scenario: Fuzz Candid parser
- **WHEN** arbitrary bytes are provided to the `candid_parser` fuzz target
- **THEN** it attempts to decode the bytes as `IDLArgs` using `from_bytes_with_config`
- **AND** a decoding quota of 2,000,000,000 is set to limit resource usage
- **AND** if decoding succeeds, `get_types()`, `to_bytes()`, and `to_string()` are called
- **AND** any errors during these operations are silently caught (no panic)

#### Scenario: Fuzz Candid type decoder
- **WHEN** arbitrary bytes are provided to the `candid_type_decoder` fuzz target
- **THEN** it attempts to decode Candid-typed data
- **AND** verifies that the decoder handles malformed input without panicking

### Requirement: Stable Structures Fuzzers

Fuzz targets in `rs/fuzzers/stable_structures/fuzz_targets/` test IC stable data structures under random operations.

#### Scenario: Fuzz stable structures multiple operations with persistence
- **WHEN** a sequence of `StableStructOperation`s is generated by the fuzzer
- **THEN** operations are executed on thread-local stable data structures:
  - `StableBTreeMap` (bounded and unbounded variants)
  - `StableMinHeap` (bounded)
  - `StableVec` (bounded)
- **AND** operations include: BTreeMapInsert, BTreeMapRemove, BTreeMapPopFirst, BTreeMapPopLast, MinHeapPush, MinHeapPop, VecPush, VecPop
- **AND** a `MemoryManager` with `VirtualMemory` manages memory allocation
- **AND** all operations are recorded for crash reproduction

#### Scenario: Crash reproduction with persistent state
- **WHEN** a panic occurs during fuzzing of stable structures
- **THEN** a custom panic hook dumps all previously executed operations to a file
- **AND** the memory state is also persisted
- **AND** this allows replay of the exact sequence that triggered the crash

#### Scenario: Key and value constraints
- **WHEN** values are inserted into bounded structures
- **THEN** the key size is fixed at 4 bytes
- **AND** values are constrained by `MAX_VALUE_SIZE` defined in the `data` module
- **AND** `BoundedFuzzStruct` and `UnboundedFuzzStruct` represent the two value types
