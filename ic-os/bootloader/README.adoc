= Bootloader

Bootloader is built somewhat similarly to the Ubuntu system, but is
kept separately (the bootloader is never changed during system upgrades).

The mutable A/B boot decision lives in +grubenv+, which records:

- +boot_alternative+: which slot (+A+ or +B+) should be tried next
- +boot_cycle+: whether that slot is already trusted (+stable+) or still in a
  probation/rollback-sensitive phase (+install+, +first_boot+,
  +failsafe_check+)

This split is intentional: upgrades rewrite inactive boot/root partitions and
then update only +grubenv+ to point at the candidate slot. The bootloader code
itself remains unchanged across normal upgrades.

On boot, the grub config runs the +boot_cycle+ state machine before loading
the kernel:

- +install -> stable+
- +first_boot -> failsafe_check+
- +failsafe_check -> switch to the opposite slot and mark it stable+

The candidate slot becomes truly +stable+ only when the booted OS later
confirms the boot by writing back to +grubenv+ (for example via
+manageboot.sh confirm+).

For GuestOS specifically, +guestos_grub.cfg+ is primarily for standalone or
custom-host boots (for example cloud deployments). When GuestOS is run by
HostOS, the normal path is direct kernel boot instead: HostOS reads the GuestOS
boot metadata and launches the kernel/initrd directly, bypassing the GuestOS
bootloader while still honoring the same +grubenv+ state machine. That normal
HostOS path is implemented in +guest_vm_runner+.

The +grub-mkimage+ utility is used to build both the EFI first stage loader
(built as +/boot/efi/EFI/Boot/bootx64.efi+, to be placed as +/EFI/Boot/bootx64.efi+
on the +efi+ partition) as well as grub loadable modules
(built in +/boot/grub/x86_64-efi+, to be placed in under +/x86_64-efi+ on
the +grub+ partition).

After this is built, the sub-directories +/boot/efi+ and +/boot/grub+ are
used to build VFAT filesystem images to go into the first two partitions
of the final disk image.
